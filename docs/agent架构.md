# 📐 架构设计 — 第1节：整体架构与 Agent Loop 核心

你说得对，Agent Loop 才是整个平台的心脏。参考 Claude Code 的核心执行模型：

```
┌─────────────────────────────────────────────────────┐
│ 客户端 (Http Client) │
└──────────────────────┬──────────────────────────────┘
 │ POST │
 ┌───────────────▼───────────────┐
 │ Agent Loop（核心循环） │
 │ │
 │ ┌─────────────────────────┐ │
 │ │ 1. Observe 接收 │ │
 │ │ 用户指令 / 工具返回结果 │ │
 │ └────────────┬────────────┘ │
 │ ▼ │
 │ ┌─────────────────────────┐ │
 │ │ 2. Think 推理 │ │
 │ │ 分析意图 / 规划步骤 / │ │
 │ │ 决定是否召唤专家智能体 │ │
 │ └────────────┬────────────┘ │
 │ ▼ │
 │ ┌─────────────────────────┐ │
 │ │ 3. Act 执行 │ │
 │ │ 选择 Tool / Skill │ │
 │ │ │ │ │
 │ │ ▼ │ │
 │ │ ┌──────────────┐ │ │
 │ │ │ 权限拦截网关 │ │ │
 │ │ │ 特危→禁止 │ │ │
 │ │ │ 高危→确认 │ │ │
 │ │ │ 通过→执行 │ │ │
 │ │ └──────┬───────┘ │ │
 │ └─────────┼──────────────┘ │
 │ ▼ │
 │ 执行结果 / 确认请求 │
 │ │ │
 │ ┌───────▼───────┐ │
 │ │ 需要继续？ │ │
 │ │ 是→回到Observe │ │
 │ │ 否→输出最终结果│ │
 │ └───────────────┘ │
 └───────────────────────────────┘
 │
 ┌───────────────▼───────────────┐
 │ 中间件层 │
 │ 会话管理 │ 记忆管理 │ 工具调度 │
 │ 技能管理 │ 持久化层 │
 └───────────────────────────────┘
```

**Agent Loop 核心：Observe → Think → Act 循环**，每次 Act 后重新观察结果，决定是否继续。权限拦截是 Act 的前置关卡，Human-in-the-Loop 在确认环节阻塞循环等待用户响应。




# 📐 架构设计 — 第2节：会话与上下文管理

会话是 Agent Loop 执行的生命周期载体。基于 HTTP通信架构，我们设计了如下机制：
**会话绑定与恢复**

- 每个 会话 连接绑定一个 `session_id`，会话元数据持久化到存储层，后续用户在同一个会话里请求，需要传入相同的session_id。
- 会话记录完整的 `Message` 列表（用户指令、智能体思考、工具调用、工具结果）。
- 当连接断开重连，或用户选择继续历史会话时，通过 `session_id` 拉取完整消息列表，实现**全量重放恢复**，确保 Agent 状态与中断前完全一致。
- 会话可绑定特定智能体（主智能体 ID 或某专家智能体 ID），恢复时自动加载对应智能体配置。
  **上下文压缩（Compact）**
- 虽然 Session 存储全量历史，但 LLM 的上下文窗口有限。
- 上下文压缩是 Agent Loop 运行时的内部行为。当历史 Token 接近阈值时，Agent 主动触发 `compact` 操作：将早期对话摘要化，保留关键指令与结果，丢弃冗余过程。
- 摘要结果仅作为当前上下文注入 LLM，**不覆盖**底层 Session 的全量历史记录。
  



# 📐 架构设计 — 第3节：记忆管理层

记忆管理为智能体提供跨越单次会话的长期认知能力。基于前期决策，本节定义核心接口与流转机制，底层存储引擎可插拔替换。
**短期记忆**
映射为当前会话的上下文窗口（最近几轮交互）。由会话管理自动维护，随 Agent Loop 的 Observe 阶段自动注入，无需额外查询操作。
**长期记忆：归档**
归档策略不交由智能体独自决定，而是通过三层控制机制实现：

1. **系统默认配置**：平台预设基础规则，如“文件操作记录保留7天”、“明确表达的偏好永久归档”。
2. **组织/项目配置**：管理员可覆盖默认规则，如“强制归档所有DB Schema变更”、“忽略所有日志类信息”。
3. **用户指令归档**：用户在交互中主动要求（如“记住这个偏好”），直接触发强制归档。
   在 Agent Loop 的 Think 阶段，系统先匹配归档规则。若信息命中规则或用户指令则自动归档；智能体自主判断归档仅在策略开关 `auto_archive_enabled` 开启时生效，否则丢弃。
   **长期记忆：查询**
   Agent Loop 在处理新任务前，调用 `recall_memory(query, top_k)` 接口。检索到的相关记忆作为上下文补充，注入到 Observe 阶段。
   **存储抽象**
   统一定义 `MemoryProvider` 接口（Archive/Recall/Delete），后续可无缝切换至向量数据库或知识图谱，现阶段可用轻量级存储占位。



# 📐 架构设计 — 第4节：工具与技能调度层

工具与技能是智能体执行的双手，两者分层解耦：工具是底层原子能力，技能是高层行为模式。
**工具层**

- **核心定义**：每个工具包含名称、描述、参数 JSON Schema 及**静态风险标签**。
- **执行机制**：核心内置工具（如文件读写、终端执行、代码搜索）在平台进程内直接运行，保证最低延迟。预留 `ExtensionToolProvider` 接口，后续可无缝对接远程 Worker、沙箱容器或外部 API。
- **注册发现**：平台启动时自动扫描注册内置工具，扩展工具通过接口动态注册。
  **技能层**
- **核心定义**：技能本质是“提示词模板 + 工具组合 + 执行约束”的集合。它不直接执行，而是指导智能体如何组合使用工具。
- **渐进式加载**：主智能体启动时仅加载轻量级的技能清单（名称 + 一句话简介）。在 Agent Loop 的 Think 阶段，智能体根据用户任务语义匹配相关技能，按需拉取该技能的完整提示词与约束定义，注入到后续的上下文中。
  **调度流转**
  在 Agent Loop 中，Think 阶段决定调用路径：
1. 优先匹配 Skill：若命中，加载完整 Skill 定义，按其模板组装工具调用链。
2. 直接调用 Tool：若无匹配 Skill，智能体直接选择原子工具执行。
   选定后进入 Act 阶段，先经过权限拦截网关，通过后再调度执行。
   

# 📐 架构设计 — 第5节：权限管理与 Human-in-the-Loop

权限与人工干预是保障企业级平台安全运行的关键护栏，贯穿于 Agent Loop 的执行链路中。
**风险矩阵管控**
基于工具/技能的静态风险标签和会话风险级别，构建二维管控矩阵：

- **特危操作**：全局硬拦截，直接返回禁止执行错误，不可豁免。
- **高危操作**：当会话级别不足以自动放行时，触发确认拦截。
- **中/低危操作**：按会话级别（如 Bypass 可自动放行中危）或直接放行。
  **拦截网关流程**
  在 Agent Loop 的 Act 阶段，工具/技能被调度前必须经过权限拦截网关：
1. 读取该操作的风险标签与当前会话级别。
2. 对照矩阵判定：通过则执行；禁止则阻断并提示；需确认则挂起当前执行流。
   **Human-in-the-Loop (HITL) 交互**
   在 WebSocket 架构下，确认拦截通过异步消息流实现：
3. **挂起与请求**：Agent Loop 暂停，通过 WS 向客户端推送 `ConfirmationRequest`（包含操作详情、风险原因）。
4. **用户决策**：客户端渲染确认交互界面，用户选择“批准/拒绝/修改参数”。
5. **恢复与回注**：客户端发送 `ConfirmationResponse`，Agent Loop 恢复执行。若批准则继续 Act；若拒绝则返回安全提示并重新进入 Think 阶段。
   此外，当智能体在 Think 阶段遇到高度不确定性问题（如需求模糊）时，也可主动触发 HITL 请求用户指引，而不仅限于风险拦截。
   

# 📐 架构设计 — 第6节：多智能体协作机制

在“主智能体 + 按需引入专家智能体”的架构下，主智能体是唯一的编排者，专家智能体是被调度的执行者。
**专家智能体的定位**
专家智能体拥有独立的 System Prompt、专属的 Tools/Skills 白名单和独立的风险边界。例如：数据库专家只暴露 SQL 执行工具，代码审查专家侧重 AST 和 Git 工具。
**召唤与委派流程**
在 Agent Loop 的 Think 阶段，主智能体若判断当前任务需要专业能力，执行以下动作：

1. **发起委派**：主智能体将“召唤专家”作为一个特殊的内部 Act，指定目标专家类型及子任务目标。
2. **上下文传递**：主智能体从当前会话中提取相关上下文（而非全量历史），打包作为子任务的输入，避免专家被无关信息干扰。
3. **专家独立循环**：专家智能体启动自身的 Agent Loop（Observe-Think-Act）执行子任务，期间产生的工具调用和 HITL 请求，由平台自动代理回当前 WS 连接。
4. **结果回传**：专家智能体完成任务后，返回执行结果或状态报告，主智能体将结果整合，继续自身的 Agent Loop。
   **状态与权限继承**
   专家智能体的执行受当前会话的风险级别约束，且主智能体需对专家的最终行为承担编排责任。



# 📐 架构设计 — 第7节：WS 交互协议与消息模型

由于我们采用 WebSocket 替代 Shell 交互，定义清晰的消息协议是客户端（终端/IDE）正确渲染和交互的基础。所有消息采用 JSON 格式，通过 `type` 字段区分。
**客户端 → 平台**

- `user_message`: 用户输入的自然语言指令。
- `hitl_response`: 对确认拦截的响应（批准/拒绝/参数修改）。
- `interrupt_signal`: 用户主动中断当前 Agent Loop（类似 Ctrl+C）。
  **平台 → 客户端**
- `agent_text_stream`: 智能体思考与回复的文本流（分片推送，支持打字机效果）。
- `tool_call_event`: 工具/技能调用事件（包含调用ID、名称、参数），客户端可据此渲染执行面板。
- `tool_call_result`: 工具执行完成的结果（成功/失败/输出内容）。
- `hitl_request`: 需要用户确认的拦截请求（包含操作详情、风险等级、原因）。
- `expert_delegation_event`: 主智能体召唤专家的事件，客户端可据此切换UI上下文（如显示"数据库专家接管中"）。
  **状态流转同步**
  每次 Agent Loop 的状态变更（Observe → Think → Act），平台均可推送 `loop_status_update`，客户端据此展示当前智能体处于"接收中"、"推理中"还是"执行中"，提升交互透明度。
  

# 📐 架构设计 — 第8节：韧性调度管道

需要将容错抽象为一个统一的横切模块：**韧性调度管道**。
**模块定位**
该管道位于 Agent Runtime 与底层执行/外部调用之间，作为所有“不确定性动作”的必经拦截层。无论是 LLM 调用、工具执行、还是 WS 推送，都不直接执行，而是提交给管道。
**声明式执行策略**
各业务模块不再自行实现 try-catch 或 retry，而是为动作声明韧性需求：

- `LLM请求`：声明 `retries=3, fallback=降级模型`。
- `工具执行`：声明 `timeout=10s, retry_on=TimeoutError`。
- `HITL等待`：声明 `timeout=30m, on_timeout=安全拒绝`。
  **管道核心能力**
1. **统一超时与重试**：管道接管执行线程，根据声明强制超时控制和指数退避重试。
2. **降级熔断**：若某类工具或 LLM 连续失败，管道触发熔断，后续请求自动走降级逻辑或快速失败。
3. **异常转译**：捕获所有底层网络或系统级异常，统一转译为平台标准错误事件，安全回注到 Agent Loop 的 Observe 阶段。
   
   通过声明式管道，容错逻辑被完全收敛为独立模块，业务模块只关注核心逻辑。
